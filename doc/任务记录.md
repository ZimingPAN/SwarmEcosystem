- 排查 time head 的监督目标与评估口径，定位物理时间学习失败的高置信根因。

### 关键结果

| 模型 | v9 overall mean | v10e_fix overall mean | 结论 |
| --- | ---: | ---: | --- |
| MuZero | +2.2215 | +0.2698 | 明显回退 |
| Dreamer | +2.3155 | +3.2298 | 未回退，反而提升 |

### 高置信结论

1. MuZero v10e_fix 的回退是真实训练退化，不是评估脚本误差。
   - 低密度 reward eval 从 v9 的 +2.2215 降到 v10e_fix 的 +0.2698。
   - Dreamer 同期从 +2.3155 升到 +3.2298，说明环境与评估口径本身没有整体失真，问题主要集中在 MuZero 路径。

2. MuZero 回退首先表现为配置漂移，而不是单一代码 bug。
   - 报告中的 v9 MuZero 配置是：max_episode_steps=200、collect_episodes=2、train_steps_per_collect=30、batch_size=64、lr=1e-4、num_simulations=16。
   - 当前训练脚本默认值是：max_episode_steps=50、collect_episodes=4、train_steps_per_collect=50、batch_size=128、lr=3e-4、num_simulations=32。
   - 这不是小调参，而是整套训练动力学一起变化，v10e_fix 与 v9 已经不是同一实验条件。

3. MuZero 训练目标本身也偏脆弱，会放大配置漂移带来的退化。
   - train_muzero_standalone.py 中 policy loss 仍然是“action taken 的 one-hot 监督”，本质更像行为克隆式目标，而不是完整的 MCTS policy 分布监督。
   - 这种目标在超参变化后更容易退化成低信号训练，难以稳定复现 v9 的规划收益。

4. 当前 time head 之所以学不到“正确物理时间”，根因是目标本身对确定性头不可识别。
   - 训练监督直接使用 sampled delta_t = -log(r) / Gamma_tot。
   - 其中随机数 r 对模型不可见，而 time head 现在只看状态 latent，不看 r，也不输出分布。
   - 因此 deterministic head 不可能从同一状态唯一恢复单次采样到的 delta_t，只能学到一个失配的平均量或塌缩常数。

5. 当前 time alignment eval 进一步把这个问题放大了。
   - eval_time_alignment.py 在 env.step(action) 之前，用当前状态先做一次 deterministic dt prediction。
   - 然后拿它去对比 step 内部新采样出来的 realized delta_t。
   - 这相当于把“状态统计量预测”与“单次随机实现值”直接逐步对齐，近零相关和量纲漂移是预期现象，不是偶然噪声。

### 时间头失败的量化证据

| 模型 | real mean delta_t | pred mean delta_t | pred/real | log corr |
| --- | ---: | ---: | ---: | ---: |
| MuZero | 3.18e-05 | 1.11e-02 | 348.7x | -0.0266 |
| Dreamer | 3.61e-05 | 3.92e-03 | 108.5x | -0.0037 |

### 关键代码证据

- LightZero-main/zoo/kmc/train_muzero_standalone.py
  - delta_t 采样：delta_t = -log(rand) / total_rate
  - policy loss：action taken 的 one-hot 监督
  - time loss：log_time_target = log(sampled delta_t)
  - 默认训练配置：50 step episode、4 collect episodes、50 train steps、batch 128、lr 3e-4、MCTS 32
- dreamer4-main/train_dreamer_standalone.py
  - 同样使用 sampled delta_t 作为 log-space 回归目标
  - 默认训练配置也已漂移到 50 step、4 collect episodes、50 train steps、batch 128、lr 3e-4
- eval_time_alignment.py
  - 当前口径是“先用当前状态预测 dt，再与执行一步后采样出来的 realized dt 对比”

### 下一步

1. 先严格复现实验条件，把 MuZero 训练配置退回到 v9 口径，再判断算法改动是否真的导致回退。
2. 把 time head 的监督目标改成状态可识别量，例如 log(1 / Gamma_tot)、E[delta_t | s] = 1 / Gamma_tot、E[log delta_t | s] = -log(Gamma_tot) - gamma_Euler，或直接预测指数分布 rate。
3. 重写 time alignment eval，使预测目标与对比目标保持一致，避免再用 deterministic state prediction 去逐点对齐 sampled realization。

### 已确认并修复的高置信根因

1. projection 性能瓶颈：
   - 远端 conda 环境没有 `scipy.optimize.linear_sum_assignment`，导致 `project_types_by_inventory()` 退回 Python greedy fallback。
   - 旧实现对整块 patch 上所有 Cu / vacancy 做 transport matching，单个 batch 的 projection profiling 达到 `638s`。

2. 验证集退化为全零：
   - `KMCEnv.reset()` 实际只清时间/统计，不会重新随机初始化晶格。
   - 旧代码在同一个 `MacroKMCEnv` 上先采 train、再采 val，会让第二次 `_collect_segments()` 落入全零验证集：`changed_mean=0`、`tau_exp≈4e-13`、`reward_sum=0`。

3. no-op 宏段污染时间监督：
   - 宏段中存在一批 `start == end` 的 no-op 样本，`changed_mask.sum()==0`，同时 `tau_exp` 近零。
   - 这类样本与 sparse lattice edit world model 的建模目标不一致，会把时间头往近零方向拉坏。
   - 已在数据集构造中跳过 no-op segment，并记录 `skipped_noop`。

4. rollout 重置无效：
   - 因为 `KMCEnv.reset()` 不重采晶格，之前所有“reset rollout”的逻辑本质上都还是在同一条轨迹上继续。

## 2026-04-03 v13_fixinit 后程回退根因定位与局部修复

### 新锁定的根因

在对 `best_model.pt` 与 `final_model.pt` 做逐样本统计后，这轮把“后程为什么重新回退到 violation=1.0”进一步钉成了一个明确的 budget/count mismatch：

1. `final_model.pt` 上，type 头并没有学出可用的 typed swap 倾向，统计表现为：
   - `mean_pred_type_change_count = 0.0`
   - `mean_vac_to_fe = 0.0`
   - `mean_vac_to_cu = 0.0`
   - `mean_fe_to_vac = 0.0`
   - `mean_cu_to_vac = 0.0`
2. 但旧训练口径里的 count surrogate 仍然在推高 `sum(sigmoid(change_logits))`：
   - `best_model.pt` 上旧 count 均值约 `0.50`
   - `final_model.pt` 上旧 count 均值约 `12.15`
3. 旧 projector 的 budget 正是直接从这项 `sum(sigmoid(change_logits))` 派生，因此 final 会被硬截到 `budget = 8`，即使它根本没有任何 typed vacancy-atom pair 支撑。
4. 结果就是：**模型其实是在“全 copy + diffuse change mass”这个局部最优里打转，但旧 projector 会把这种状态误判成 violation=1.0。**

### 已落地修复

本轮对 `dreamer4-main/dreamer4/macro_edit.py` 与 `dreamer4-main/train_dreamer_macro_edit.py` 做了两处最小修复：

1. projector 的 budget 不再直接依赖 `sum(sigmoid(change_logits))`，而改为：
   - 用 `typed_change_mass = sigmoid(change_logits) * (1 - p(copy current type))` 估计有类型变化支撑的 change 质量；
   - 再用实际可组成的 typed pair 数量对 budget 做上界裁剪；
   - 如果根本没有 typed swap 支撑，就不再把 diffuse change mass 误报成 violation。
2. 训练里的 count surrogate 改为 `_soft_typed_change_count()`，不再奖励“把 change 概率均匀涂满整个 patch 但 type 头仍保持 copy”的解。

### 本地验证

1. 测试新增两条回归：
   - diffuse change mass + 全 copy type 不应再触发 violation；
   - `_soft_typed_change_count()` 在缺少 non-copy type 支撑时应接近 0。
2. 本地回归结果更新为：`20 passed, 2 warnings`。
3. 对已有 `final_model.pt` 直接在新代码路径下重评：
   - `reachability_violation_rate: 1.0 -> 0.0`
   - `projected_change_f1` 与 `projected_changed_type_acc` 仍为 `0.0`
4. 这说明这轮修复首先解决的是“错误的 violation 口径”，证明后程回退里确实包含一部分 projector/count 语义 bug；但它还没有单独把 path 学习本身拉起来。

### 当前判断

1. 现在可以高置信地说，后程 `violation=1.0` 并不完全等价于“模型真的在预测大量不可达 swap”；其中一部分只是旧 budget 公式把 diffuse mask mass 误判成了 edit 预算。
2. 当前剩余问题已经进一步缩小为：为什么 type 头在后程继续保持全 copy，而 change 头却被 count 目标推向 diffuse mass。
3. 下一步如果继续推进，应该优先验证新 count surrogate 下的短程训练是否能让 typed pair 支撑重新出现，而不是再围绕旧的 violation 指标本身做判断。

### 背景

v16 训练中观察到 `mask_aux` 从 1.0 降到 0.10。经分析发现 `mask_aux` 是 `_scheduled_aux_scale()` 的预设退火调度器输出（不是模型指标），辅助 mask loss 权重从 epoch 55% 开始从 1.0 线性衰减到 0.1。

但 proj_global_l1 单调增长仍是核心问题。尝试用稀疏性约束来验证假设。

### 代码修改

1. `_edit_supervision_losses` 新增参数 `sparsity_weight` 和 `count_loss_weight`
2. 添加稀疏惩罚：`sparsity_loss = torch.sigmoid(change_logits[unchanged_valid]).mean()`
3. CLI 新增 `--no_aux_anneal`, `--mask_sparsity_weight`, `--count_loss_weight`
4. `_train_epoch` 支持 `aux_anneal=False` 保持 aux_scale=1.0

### 训练配置

GPU 7, 80 epochs, --no_aux_anneal --mask_sparsity_weight 0.3 --count_loss_weight 0.3
其余参数同 v16

### 决定性发现与结论

1. **proj_global_l1 最终值完全相同（0.049）**：无论是否使用稀疏约束，GNN encoder 演化都导致 proj_global_l1 收敛到相同水平。这彻底否定了"mask 过度编辑导致 proj_global_l1 增长"的假设。
2. **真正的根因**：GNN encoder 随训练变得更有区分力，使得投影（连续 logits → 离散原子类型 → 重新编码）产生的量化误差在 latent space 中被放大。这是 encoder 表达力增长的自然结果。
3. **稀疏约束的实际效果**：
   - ✅ mask_loss 完全稳定（2.45 vs 1.85），训练行为更健康
   - ✅ epoch 时间大幅降低（~200s vs 375-3352s），因为编辑原子数稳定
   - ✅ unchanged_copy_acc 更高（0.729 vs 0.484），保守预测更准确
   - ✅ change_f1 略好（0.522 vs 0.495）
   - ❌ 未改善 proj_global_l1 增速（最终值完全相同）
   - ❌ 最佳 score 略差（3.729 vs 3.659）
4. **保守/激进振荡**：v17 的 proj_change_f1 在 0.024-0.352 之间大幅振荡，说明 BCE 重平衡和稀疏惩罚之间存在竞争。

### 下一步方向（基于新的根因理解）

问题核心：proj_global_l1 = |GNN(projected_types) - GNN(gt_types)| 随着 GNN 表达力增长而必然增大。需要打破这个耦合。

可能的解决路径：
1. **Detach encoder for proj_global_l1**：计算 proj_global_l1 时 detach GNN encoder，用固定的 encoder 快照评估，避免 encoder 灵敏度放大
2. **降低 score 公式中 proj_global_l1 的 80× 权重**：当前权重过高，一个绝对 L1 差值增长几个百分点就能抹平所有其它指标的改善
3. **Stop-gradient projection path**：projection loss 训练不回传到 encoder，让 encoder 自由演化
4. **归一化 proj_global_l1**：用 per-site 归一化或 cosine similarity 替代绝对 L1
5. **Early stopping + fine-tune**：接受 proj_global_l1 增长是不可避免的，主动使用早期 checkpoint 作为最终模型

## v18_detach：Encoder Detach 实验（反面验证）

### 背景

v16/v17 实验表明 proj_global_l1 增长可能是 GNN encoder 区分力增强的自然结果。尝试在 `_projected_state_alignment_loss` 中 detach encoder 输出（projected_global、projected_patch_latent），阻断投影一致性梯度流向 encoder。

### 代码修改

在 `_train_epoch` 中添加 `detach_proj_encoder` 参数。当启用时，`projected_global.detach()` 和 `projected_patch_latent.detach()` 被传入 `_projected_state_alignment_loss`，proj_state_loss 不再训练 encoder。

### 训练结果

**最佳模型**：Epoch 10, score=**3.968**（远差于 v16 的 3.659）

最终 epoch 80 指标对比：

| 指标 | v16 | v17 | **v18** |
|------|:---:|:---:|:---:|
| 最佳 score | **3.659** | 3.729 | 3.968 |
| 最终 proj_global_l1 | 0.049 | 0.049 | **0.120** |
| 最终 proj_loss (训练) | 0.048 | 0.051 | **0.132** |
| 最终 mask_aux | 0.10 | 1.00 | 0.10 |
| 最终 epoch time | 3352s | 313s | **50s** |

### 决定性结论

1. **Detach encoder 是错误方向**：proj_global_l1 从 0.049 暴增至 0.120（2.5 倍），证明 proj_state_loss 对 encoder 的梯度信号是**必要的正向约束**
2. **proj_state_loss 的实际作用**：让 encoder 学习"投影状态的表征应接近真实状态的表征"，这个一致性约束**减缓**而非加速 proj_global_l1 增长
3. **原始假设被推翻**："encoder 灵敏度放大导致 proj_global_l1 增长"的分析部分正确，但解决方向错误。detach 移除了唯一的约束力，反而加速了分歧
4. **训练极快（50s/epoch）但质量极差**：因为 proj_state_loss 不产生 encoder 梯度，计算量大幅减少

### 三版本综合对比与反思

| 实验 | 假设 | 结果 | 学到的教训 |
|------|------|------|-----------|
| v16 (基线) | 向量化优化后 proj_every=1 可行 | ✅ 确认 | 代码性能不是瓶颈 |
| v17 (稀疏) | mask 过度编辑导致 proj_l1 增长 | ❌ 否定 | proj_l1 增长与 mask 行为无关 |
| v18 (detach) | encoder 灵敏度放大导致 proj_l1 增长, detach 可解决 | ❌ 否定 | proj_state_loss 是必要约束, detach 加剧问题 |

### 仍有效的解决路径

以上三个实验排除了两个方向。剩余可行路径：
1. **降低 score 公式中 proj_global_l1 的 80× 权重**：允许后期 epoch 的 tau/path 改善体现在 score 中
2. **归一化 proj_global_l1**：用 cosine similarity 或 relative L1 替代绝对 L1
3. **加强 proj_state_loss 权重**：既然这个约束有用，加大 weights["proj"] 从 0.5 到 1.0-2.0 → **已在 v19 中验证**
4. **分阶段训练**：Phase 1 只学 encoder+dynamics（不含 projection），Phase 2 加入 projection 微调
5. **接受早期停止**：v16 的 best_model.pt (ep10, score=3.659) 就是当前最佳模型

## MuZero v18：梯度隔离修复

### 问题诊断

v16b/v17 性能崩溃的根因：**time_loss 未做梯度隔离，违反 AGENTS.md §6 rule 5**

1. `latent` 未 `.detach()`，time_head 的大梯度回传到 representation network
2. `time_loss` 权重 1.0（与 policy_loss 等权），v17 中 t_loss 初始高达 25-33，完全淹没策略梯度
3. v9 根本没有 time_loss（纯策略学习），所以 v9 不受影响

### 修复内容

修改 `LightZero-main/zoo/kmc/train_muzero_standalone.py`：

```python
# 修复前（有问题）
log_time_pred = model.predict_log_time_delta(latent)       # ← 无 detach
loss = policy_loss + reward_loss + 0.25 * value_loss + 1.0 * time_loss  # ← 混合

# 修复后（符合 AGENTS.md）
log_time_pred = model.predict_log_time_delta(latent.detach())  # ← 梯度隔离
# 两次独立 backward：main_loss → backbone, time_loss → 仅 time_head
```

### 训练配置

- 保持 v9 默认参数：max_episode_steps=200, lr=1e-4, batch_size=64, num_simulations=16
- GPU：cuda:4（CUDA_VISIBLE_DEVICES=4）
- tmux 会话：`muzero_v18`
